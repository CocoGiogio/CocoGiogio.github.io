Directory Enumeration using FuFf


      ffuf -u http://10.10.247.21:80/FUZZ \
        -w /usr/share/wordlists/SecLists/Discovery/Web-Content/big.txt \
        -e .php,.txt,.rsa,.conf,.bak,.env,.json,.old,.ini \
        -fc 404,403

Using this command we will fuzz the end of the URL. We will be searching for directories, or file extensions that end in .php, .txt, .rsa, .conf, .bak, .env, .json, .old, .ini and exclude any page that have a response code of 404 or 403

After reading the contents of this exploit using the ?urlConfig parameter, we can leverage it for LFI and RFI. We can read the contents of /etc/passwd with a payload like:

It is also possible to have the server fetch a remote PHP shell by pointing urlConfig at an externally hosted PHP file. The reverse shell used in testing was hosted on an attacker-controlled webserver and fetched by the target.

Example of the fetch payload used (host and ports depend on your environment):

Here we notice that the script: /home/milesdyson/backups/backup.sh gets run with root permissions. Let's look at the backup.sh script:

The tar usage in the backup process can be abused using checkpoint-action/wildcard injection to execute commands during extraction (note: check GTFObins for safer, defensive guidance and examples). In this case the test used a method that creates a small shell script in the backup root and triggers execution during the backup process.

The actions executed in the test included creating a script to append a sudoers entry for www-data, then creating the magic "checkpoint-action" file to trigger it during tar extraction. After that, the environment allowed privilege escalation to root and the root flag was retrieved. 


      # Example of the files created (paths depend on target and backup folder)
      echo 'echo "www-data ALL=(root) NOPASSWD: ALL" >> /etc/sudoers' > sudo.sh
      touch "/var/www/html/ - checkpoint-action=exec=sh sudo.sh"
      touch "/var/www/html/ - checkpoint=1"

After obtaining sudo rights for www-data, root access was used to retrieve the root.txt flag.
